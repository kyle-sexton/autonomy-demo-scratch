# Repo-specific identifier deny-list for portability test v2.
#
# Per `.claude/rules/skill-behavior-layering.md` "Worked examples / advanced patterns" → "HARD-FAIL vs WARN tiering (portability deny-list)".
#
# Format: one identifier per line. Prefix:
#   H: <token>     — HARD-FAIL: portability test exits 1 if token appears in SKILL.md / BEHAVIOR.md / behavior/<*>/default.<ext>
#   W: <token>     — WARN: portability test emits `WARN:` line but still exits 0
#   # comment      — header / section divider, ignored by scanner
#
# Tiering rationale (per `skill-behavior-layering.md` "HARD-FAIL vs WARN tiering"):
#   - HARD-FAIL = unambiguous repo-specifics. No other repo would reuse the literal — embedment is a portability bug.
#   - WARN = stack-choice tokens. Other shops on the same stack legitimately reuse the same name (Aspire, Blazor, Duende, MSSQL).
#     Forcing every consumer to tokenize stack names would defeat the purpose; only configuration paths / project names /
#     specific version pins are repo-specific.
#
# Scope:
#   - Token matching is literal substring (no regex), case-sensitive. Patterns like `PLAT[0-9]{3}` would require regex —
#     deferred to v3 if needed. Today we use a representative set `PLAT001` through `PLAT022` enumerated.
#   - Scanned in SKILL.md + BEHAVIOR.md + every behavior/<point>/default.<ext>. `team.<ext>` and `user.<ext>` are EXCLUDED —
#     team/user layers ARE the home for repo-specifics by design.
#
# Per `agent-trust.md` "Repo-scoped vs contributor-scoped state": rules describe the surface (file path / analyzer code is
# repo-scoped; stack name is industry-scoped), not the contributor.
